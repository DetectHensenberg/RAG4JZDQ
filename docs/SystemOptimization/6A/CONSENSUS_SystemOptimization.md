# CONSENSUS: System Optimization (A4-A6)

## 1. 明确的需求描述与验收标准
- **需求**：对 RAG 系统进行系统级并发优化，解决数据库锁问题与同步 I/O 阻塞。
- **验收标准**：
  - [ ] 产出压测对比报告，P95 响应时间在并发 5 下提升 > 30%。
  - [ ] `database is locked` 错误在并发写场景下不出现。
  - [ ] 代码库核心检索链路无 `os.path` 与 `open()` 的同步调用。

## 2. 技术实现方案与集成
- **方案**：
  - A4: 自研类 Stdio 压测脚本，模拟多并发 Stdin 注入。
  - A5: 引入 `SQLiteManager` 统一 WAL 模式与 20s busy_timeout。
  - A6: 对 DB 读写使用 `asyncio.to_thread` 封装，对文件操作引入 `anyio`。

## 3. 技术约束与集成方案
- **资源利用**：压测期间不限制 CPU 核心使用，但需监控内存泄漏。
- **版本控制**：优化过程中不引入对 `aiosqlite` 等大改动三方库的破坏性依赖，优先基于 `sqlite_utils` + `to_thread` 实现。

## 4. 任务边界限制
- 仅优化本项目的 SQLite 数据库，不涉及 ChromaDB 底层 C++ 层的优化。
- 仅优化 `src/core` 下的核心链路，不涉及对 `tests` 脚本本身的异步重构（压测脚本除外）。

## 5. 最终确认
- 所有不确定性已解决。
- 技术路线对齐现有稳定性调优目标。
