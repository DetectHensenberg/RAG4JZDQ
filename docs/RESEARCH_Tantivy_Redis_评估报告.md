# 专题报告：Tantivy 稀疏检索与 Redis 异步摄取评估

本报告针对主线遗留的 A7（Tantivy）与 A8（Redis）两项性能增强提案进行深度分析，旨在为后续 PoC（概念验证）提供决策依据。

---

## 1. Tantivy 稀疏检索替代评估 (A7)

### 1.1 它们是什么？
**Tantivy** 是一个用 Rust 编写的高性能全文搜索库（类似于 Lucene）。在 RAG 系统中，它作为 **Sparse Retriever (稀疏检索器)** 的底层引擎，负责基于关键词（BM25）的召回。目前我们使用的是 Python 原生实现的 `BM25Okapi`（基于 NumPy/Pickle）。

### 1.2 为什么要做这个 (动机)？
*   **内存瓶颈**：现有的 Python BM25 方案需要将整个索引加载到内存中（Pickle 序列化），当文档量达到万级或十万级时，内存消耗会极具飙升。
*   **加载速度**：每次重启服务器都要重新加载巨大的 Pickle 文件，导致启动延迟。
*   **查询性能**：Python 解释器的循环在跨多字段关键词匹配时存在性能上限。

### 1.3 有什么好处？
*   **超轻量内存**：基于 Mmap（内存映射）技术，索引存储在磁盘，仅在查询时按需映射，内存占用降低 80% 以上。
*   **极致速度**：作为“世界上最快的全文搜索引擎之一”，查询延迟通常在毫秒级，且支持多线程并发查询。
*   **工业级功能**：支持增量索引更新（无需重建整个索引）、字段加权、停用词过滤等高级特性。

### 1.4 技术复杂性
*   **中等 (Medium)**：
    *   需要安装 `tantivy` Python 绑定（或通过 C-FFI 调用）。
    *   需要重构 `SparseRetriever` 的底层存储逻辑，从读写 `pickle` 切换为管理 Tantivy Index 目录。
    *   需要处理 Rust 与 Python 之间的数据转换开销。

### 1.5 有什么缺陷？
*   **环境依赖**：需要编译好的二进制轮子（Wheels），在某些受限的生产环境（如某些国产化容器）可能存在兼容性安装问题。
*   **调试难度**：如果底层索引损坏或出现 Segfault，Python 层的 Traceback 可能不够直观。

## 2. Redis 队列化摄取评估 (A8)

### 2.1 它们是什么？
**Redis** 在这里作为 **任务分发中心 (Message Broker)**。目前我们的 Ingestion（摄取）是直接在 API 进程中通过 `asyncio` 并发的。改为 Redis 后，API 仅负责把文件“入队”，由独立的 Worker 进程异步处理解析、切片及向量化。

### 2.2 为什么要做这个 (动机)？
*   **防止 API 崩溃**：大规模上传文件（如 100 份 PDF）时，PDF 解析和向量化是 CPU 密集型操作，会阻塞 API 响应甚至导致 OOM（内存溢出）。
*   **持久化保证**：如果上传中途服务器重启，当前内存中的任务会丢失；Redis 队列可以持久化任务，重启后自动续传。
*   **水平扩展**：可以通过增加 Worker 容器的数量轻松提升摄取吞吐量。

### 2.3 有什么好处？
*   **解耦**：摄取压力与查询压力完全隔离，互不干扰。
*   **可观测性**：可以清晰地看到“待处理”、“处理中”、“成功/失败”的任务队列状态，支持重试策略。
*   **削峰填谷**：面对瞬时高频上传，系统能按设定的速率平滑处理，不会冲垮底层向量数据库。

### 2.4 技术复杂性
*   **中高 (Medium-High)**：
    *   需要引入 Redis 服务作为外部依赖。
    *   需要集成任务队列框架（如 `RQ` 或精简的自定义 `Redis Stream` 逻辑）。
    *   需要处理跨进程的“摄取进度”同步问题（目前是通过 SSE/Memory，未来需通过 Redis Pub/Sub 或状态同步）。

### 2.5 有什么缺陷？
*   **运维成本**：多了一个 Redis 组件需要部署、备份和监控。
*   **架构复杂性**：引入了异步一致性问题（用户上传后，查询可能需要等待几秒钟才能搜到最新内容）。

---

## 3. 总结建议

| 特性 | Tantivy (检索优化) | Redis (摄取加速) |
| :--- | :--- | :--- |
| **解决核心痛点** | 慢启动、高内存占用 | 上传卡顿、不可靠续传 |
| **主要收益** | 检索延迟降低、单机容量提升 | 系统伸缩性、任务持久性 |
| **建议优先级** | **P1 (高)**：建议优先开始 PoC | **P2 (中)**：规模化需求明确后再落地 |

**结论**：如果您之后计划向公网开放或处理 **GB 级** 以上的文档库，**Tantivy** 是必须跨过的技术红利；如果您更在意 **海量文件上传时的系统抗压能力**，则 **Redis** 是架构升级的首选。
