# SOLUTION - Antigravity 全局能力白名单

> 日期：2026-03-23  
> 目标：定义跨项目通用的个人底座能力清单，供所有项目（包括新项目）挂载和复用。

---

## 1. 定位说明

全局能力白名单指的是 Antigravity (Agent) 默认开启的、不与特定业务耦合的底层能力。无论在哪个目录下工作，都应遵循这些通用最佳实践和编码约束。

## 2. 核心白名单清单 (V1)

以下技能（Skills）将被配置为全局默认外挂能力（按重要优先级排序）：

### 研发底座与规则约束
- **`search-first`**：避免“重复造轮子”。要求 Agent 在开始生成代码、调用或实现某功能前，先搜索现有工程、业务模式或者外部成熟方案。
- **`verification-loop`**：构建“设想 -> 执行 -> 验证”闭环。确保编写的任何逻辑或脚本必须由 Agent 直接给出验证方法及初步自测结果。

### 研发安全底线
- **`security-review`**：提供安全代码检查指南、防范注入与越权等操作参考。由于涉及到底层数据与隐私安全，所有项目级别的开发中均应具备该全局底线意识。

### 语言专项标准（以 Python 为例）
- **`python-patterns`**：设定全局通用的 Pythonic 命名、类结构、类型提示规范（Type Hints, PEP-8）。
- **`python-testing`**：建立以 `pytest` 作为单元测试运行器、合理使用 Fixtures、Parametrize 及 Mock 技术的底层测试共识。

### 预留扩展池
- **`api-design`** 与 **`backend-patterns`**：适用于跨项目的后端或中台开发工作。但在轻量化或纯前端项目中可作静默处理，属于“按需就绪（On-Demand）”状态。

## 3. 全局 Hooks 配置策略说明

在当前的全局形态下，必须遵循安全、无痛的原则，防止对未知工程造成破坏：

1. **禁用静默的自动写入**：禁止全局层面的强自动化修改（例如自动 `auto-fix` 或未经确认的全局代码替换等）。修改代码需要通过明确的任务状态反馈给人工，并获准后才能自动化执行。
2. **启用警告与指导型提示**：在保存、提交或执行全局检查时，主要抛出 `ruff` / `mypy` 等工具的提示报告，并将报告以及修复建议汇总反馈给开发者，最终由人来做执行的决定。
