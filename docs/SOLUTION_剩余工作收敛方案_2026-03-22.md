## 3. 执行策略（6A 工作流）

以下按 6A 工作流推进，先主线后分支。

### A1 - Align（对齐）

1. 冻结主线范围：仅执行“主线必做 A 组 1-8”。
2. 标书助手与前端替换任务进入“待排期池”。
3. 明确验收口径：每项必须有测试或观测指标。

### A2 - Architect（设计）

1. 交互增强设计：Thinking 状态机、反馈数据模型、追问生成协议。
2. 性能治理设计：压测方案、SQLite 参数矩阵、I/O 扫描清单。
3. 预研设计：Tantivy 与 Redis 仅做 PoC 评估，不直接替换生产链路。

### A3 - Atomize（原子拆解）

按两个 Sprint 拆解：

1. Sprint-1（体验闭环）
   - Thinking 阶段化展示
   - 反馈回路全链路
   - Suggested Questions
2. Sprint-2（性能与可扩展）
   - 压测基线
   - SQLite 参数调优
   - 同步 I/O 清理
   - Tantivy/Redis 双 PoC

### A4 - Act（实施）

1. 后端优先：先保证 API/数据模型可用，再接前端。
2. 每项改动配对应测试（单测/集成/压测脚本）。
3. 所有改动记录到 `docs/context.md` 与本方案附录。

### A5 - Assert（验证）

1. 功能验收：
   - Thinking 展示可见且与真实阶段一致
   - 反馈可入库、可查询、可统计
   - Suggested Questions 可稳定生成
2. 性能验收：
   - 给出改造前后延迟/QPS对比表
   - SQLite 参数选择有数据依据
3. 稳定性验收：
   - 关键回归测试全绿
   - 不引入新阻塞点

### A6 - Archive（归档）

1. 产出 `FINAL_剩余工作收敛.md`（结果、指标、风险、遗留）。
2. 更新 `系统审查与优化报告.md` 的状态标记。
3. 将“历史分支待排期”迁移至独立路线图文档。
