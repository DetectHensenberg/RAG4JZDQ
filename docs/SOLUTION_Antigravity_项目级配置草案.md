# SOLUTION - Antigravity 项目级配置草案 (RAG)

> 日期：2026-03-23  
> 目标：基于全局环境能力之上，定制化适用于当前 RAG 本地代码工程的专属 Agent 能力与约束机制。

---

## 1. 定位说明

由于 RAG 属于偏向模型能力和链路处理的后端工程化系统。此配置作为对 `everything-claude-code` 架构的“剪裁与增强”，用以建立专门面向该仓库日常迭代特化的快捷入口和行为准则。

## 2. 专属 Skills（业务特化能力库）

基于本地 `<available_skills>` ，该仓库当前需要优先绑定的核心项目级技能：

- **`qa-tester` (项目特化版)**: 具备理解本项目下 `QA_TEST_PLAN.md` 逻辑的能力，能自动贯穿 Streamlit、MCP、CLI 等各个交互切面进行回归测试与断言。
- **`auto-coder` (任务驱动特化)**: 衔接 `DEV_SPEC.md` 要求与本项目下的 `6A 工作流`（Align -> Architect -> Atomize -> Approve -> Automate -> Assess），完成结构化的闭环编码开发。

## 3. 核心 Commands 入口（标准化钩子）

为了约束流程不随意散落，需提供以下统一的可复用入口，支撑起敏捷化的 Agent 节点工作模式：

1. **`Run QA` (测试门禁入口)** 
   - 触发动作：全面执行基于本项目业务用例的回归与性能测试（覆盖 Retriever / Ingestion 队列 / 解析链路等）。
   - Agent 行为：运行脚本，实时捕捉日志和失败堆栈，若遇到非预期内失败则主动提报并协助修复。

2. **`Sync Docs` (文档对齐入口)**
   - 触发动作：模块间解耦、重构、或对外暴露的类与方法发生字段级及以上变动。
   - Agent 行为：通过代码逻辑反向推演，主动去更新本项目的 `docs/context.md`，或相关的 `DESIGN_*.md` 结构文件。坚决规避“实物与描述不符”。

3. **`Release Check` (代码与安全审计入口)**
   - 触发动作：准备合入代码、节点提测前。
   - Agent 行为：链式调用项目内置自动化任务管道，如 `pytest --cov`、`ruff check` 等，形成整体质量闭环评估。

## 4. Hooks 管控与限制方案

本项目的自动化干预权限应建立在**可信赖**的基础之上，策略如下：

### 4.1. 提醒 / 只读型 Hooks —— **全量开启**
- 检查是否存在新增依赖但是未同步记录。
- 检测模块内的 Docstring 是否为空。
- 若 `API KEY` 等敏感常量出现在非 `.env` 位置，提出红线警告阻断。

### 4.2. 自动写入 / 执行 Hooks —— **降权控制（限制执行区域）**
- **严禁**：未经用户明确准许的大规模全局替换和模块物理迁移。
- **允许**：在单独的某个 Task 生命周期内，经过 `Approve（审批阶段）`，只对划定的目标边界范围内的目标文件执行并发的 CRUD 。

## 5. 多人协作规范与 6A 模版

项目的架构体系需要长期收敛。在后续多人协同下，每一次核心特写开发将遵循 `6A工作流`：
- 所产生的一切过程文档（比如 `ALIGNMENT_[].md`、`DESIGN_[].md`、`CONSENSUS_[].md`、`FINAL_[].md`）都具备统一的项目组织归档标准，均放置于 `docs/[任务名]/` 路径下，建立唯一的事实追踪来源。
