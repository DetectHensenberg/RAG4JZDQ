# DESIGN: 性能平衡版摄取流水线架构

## 1. 架构原则
针对个人 PC 环境（非工业级算力集群），设计的核心考量点是：**降低单点耗时、控制内存/显存峰值、缩减 API 调用成本，同时最大限度保留内容和图片的检索价值**。

## 2. 模块设计 (Balanced)

### 2.1 PDF 解析引擎选型 (平衡的A方案)
- **拒绝重型方案**：不启用 `DoclingPdfLoader`（需要额外下载并常驻加载超 2GB 的深度学习模型，普通 PC 显存压力过大，耗时也极高）。
- **优化选用**：继续沿用项目当前默认配置的 **`LayoutPdfLoader`**。它利用极度轻量级的 C++ 基础库 PyMuPDF (`fitz`) 进行文本坐标测算和启发式分栏判定，能在几秒内完成解析，且对多栏文献有极佳的抗干扰能力。
- **极致进化点（Image Extraction 增强）**：
  当前 `LayoutPdfLoader` 图片提取使用的是原生碎片化抽取（容易抽出几百个无用的 PDF 背景修饰线、小图标，或者抽出破碎的半张图）。我们将把原本写在 `PdfLoader` 里非常优秀的**“整页高质量渲染（Pixmap Rendering）”算法**无缝移植/合并到 `LayoutPdfLoader` 中。这样既能靠 C++ 引擎保证纯文本排版精准、秒级入库，又能保证带插图的页面展现全貌被大模型完整视觉识别（解决之前只有 1 张碎图的系统遗憾）。

### 2.2 提纯与富化策略 (平衡的B/C方案)
- **拒绝深渊开销**：在入库环节（后台批量静默阶段）**彻底关闭**大模型级别的 Chunk 重写 (`use_llm: false`) 和全局 GraphRAG 实体穷举。
- **轻量级替代**：
  - 依赖纯文本字面规则剔除死链和乱码（流水线中已存在 `ChunkRefiner` 规则分支逻辑）。
  - 在分块器 `DocumentChunker` 中继续保持自带的“Markdown 标题层级感知”，通过正则嗅探 `#标题`，将上下文位置写入元数据 `heading_path`。实现**零 Token 消耗、零毫秒延迟的上下文防割裂挂载**。
