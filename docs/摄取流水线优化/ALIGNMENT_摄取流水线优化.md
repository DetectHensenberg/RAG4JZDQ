# ALIGNMENT: 摄取流水线优化

## 1. 原始需求
用户询问：“摄取流水线是否还有优化空间？比如输入的PDF和PPTX转换成MD，再摄取？”

## 2. 需求理解与现状分析（项目上下文）
经过对 `src/ingestion/pipeline.py` 及 `src/libs/loader/` 的源码分析，系统**当前已经完全采用了“先转 Markdown 再摄取”的先进架构**：
- **PDF 处理**：`PdfLoader` 默认底层调用微软的 `MarkItDown` 库，先将 PDF 页面转成结构化的 Markdown。如果页面含有图片，还会提取原生图片并插入 `[IMAGE: xxx]` 的 Markdown 占位符。
- **PPTX 处理**：`PptxLoader` 支持将幻灯片渲染成图片后，交给 Vision LLM大模型“看图说话”，直接输出包含标题、表格和演讲者备注的丰富 Markdown 格式文本。
- **Markdown 感知分块 (Chunking)**：`DocumentChunker` 读取 MD 文本，在分块时不仅切分，还能识别 `#` 标题层级，自动维护 `heading_path`（如“第一章 > 第二节”），并且能自动识别 markdown 的表格与代码段。

## 3. 真正的优化空间 (智能决策策略)
既然“转 MD”和“Markdown 感知”已经是现有基建，摄取流水线的深度优化空间主要集中在**“如何把 MD 转换得更精准”**以及**“高级预处理机制的启用”**：

### 优化方向 A：升级 PDF 解析引擎（解决复杂排版与表格）
轻量级的 `markitdown` 对付单栏文本很好，但遇到复杂多栏排版、跨页且带有合并单元格的表格时，生成的 MD 会错乱。
- **优化方案**：源码 `loader_factory.py` 中实际已经预置了对接 `DoclingPdfLoader` (IBM Docling 深度学习版面分析) 和 `LayoutPdfLoader` 的口子。你可以通过将配置项引擎无缝切换为这类强力的模型驱动引擎，彻底解决由“垃圾进”导致的“垃圾出”问题。

### 优化方向 B：开启大模型富化 (Metadata & Context Enrichment)
- **优化方案**：当前流水线的 Transform 管道内置了大模型润色 (`ChunkRefiner`) 和打标 (`MetadataEnricher`)。可以优化这些 Prompt 使其强制执行“假设性问题生成（HyDE）”或者“给子 Chunk 自动拼接父标题上下文”，极大地防范内容在切片后的语义丢失。

### 优化方向 C：开启父子分块与 GraphRAG
- **优化方案**：`pipeline.py` 中有两行沉睡的高级代码：`HierarchicalChunker` (父子层级防割裂) 和 `GraphExtractor` (知识图谱与 GraphRAG)。当你的硬件吃得消时，可以通过配置项打开它们，让流水线在存入纯向量前，额外构建一张语义知识图谱网络。

## 4. “三管齐下”（ABC 全部开启）的系统代价与弊端
如果你打算火力全开，同时启用 **强力版面解析大模型 (A) + 块级大模型密集重写 (B) + 全局图谱穷举提取 (C)**，必须提前知悉以下极其严重的工程后果：

1. **极其可怕的时间开销（解析耗时呈指数级爆炸）：**
   原本一份 100 页的 PDF 用 MarkItDown 只要 10 秒就能全部入库。一旦全开，Docling (A) 需要吃满 CPU/GPU 进行像素级 OCR 和布局推断（耗时几分钟）；而切分出的几百个 Chunk，每一个都要排队等待大模型执行润色改写 (B) 和实体图谱关联关系抽取 (C)，一份百页标书的入库时间可能会从**十几秒瞬间暴涨到一两个小时**。
2. **直线上升的金钱成本（Token 消耗深渊）：**
   基础版 RAG 只有在“调用 Embedding 向量模型”时耗费非常便宜的微型 Token。一旦开启 B 和 C，原本 10 万字的文档，需要与思维大模型进行成百上千次“一问一答”来重写和提取实体。单次大文件入库可能会烧掉数十万乃至上百万级别的输出 Token，硬成本极速攀升。
3. **API 限流瓶颈（Rate Limit 击穿）：**
   要在可接受的时间内处理完 B 和 C 的海量 Chunk 提纯，必须在流水线后台高并发呼叫 LLM API。这就极易瞬间触发服务商（比如 OpenAI 或 Kimi）的并发防刷限制（`HTTP 429 Too Many Requests`），导致整条流水线崩溃挂机停摆。
4. **单点故障与事务回滚难题（一旦失败就是几小时白干）：**
   当单条文件的流水线处理被强行拉长到一小时之久，如果在提取到 99% 时发生了一次大模型网络超时，前面所有的计算和耗费的 Token 极容易前功尽弃（因为目前系统只做到了文件级的完整性检查，很难做到每个 Chunk 单独的断点续传）。

*结论：ABC 全开等同于搭建了一个工业级的「重型数据提纯工厂」。对于非极端高精度场景，强烈建议日常仅选用“A（强版面拦截关键长表格）”，将 B 留给前端检索/交互时再做实时改写，或者对特定高度重要文档做局部的 C（仅核心内容建图），而不是在后台摸黑入库时为了极致的完美主义通吃海量泥沙。*
