## 2. 诊断与修复 ChromaDB HNSW 索引损坏
- **故障现象**：后端启动时不断爆出 `HNSW unhealthy: bad index_metadata.pickle` 的警告信息，导致日志杂乱且可能对向量检索造成隐藏隐患。
- **排查过程**：
  - 编写临时诊断脚本 `check_chroma.py` 遍历了 `data/db/chroma/chroma.sqlite3`，获取数据库内的 segment 表与实际磁盘的 UUID 目录进行了深度交叉比对。
  - 发现两个核心问题目录：`140879bf`（磁盘存在但 SQLite 无记录的纯孤立垃圾目录）、`67b9d803`（属于 products 集合的 VECTOR segment，但目录大小为 0，缺失了 metadata 文件）。
- **安全修复策略**：
  1. **全量数据备份**：在进行任何写操作前，使用 PowerShell 脚本将所有的 SQLite 数据库与图像库（`image_index.db`, `ingestion_history.db` 等）备份至 `data/db/backup_20260321_xxxxxx`。
  2. **清理孤立数据**：安全删除了 SQLite 无记录的 `140879bf` 垃圾废弃目录。
  3. **强制刷新空集合索引**：针对被识别但缺失 metadata 文件的 `67b9d803` (属于 products 集合)，推断其是因为集合内只有空数据、未落盘 HNSW 文件导致。因此编写了 `fix_hnsw.py` 脚本：**临时写入一条假数据（Dummy Vector）并立刻删除**，成功诱发 ChromaDB 将完整的 HNSW 结构（包含 `index_metadata.pickle`）持久化落盘。
  4. **优化健康检查脚本**：修改与完善了 `chroma_backup.py` 文件中的心跳和合规检测，使其自动跳过由 ChromaDB 对空集合预创的空目录，彻底根除“假性损坏”警告。
