## 1. PDF 多栏排版 — 版面分析 ✅ 已实现

### 实现详情

src/libs/loader/loader_factory.py             ← pdf_parser=="layout" 路由到 LayoutPdfLoader

```

**`LayoutPdfLoader`** 使用 PyMuPDF `page.get_text("dict")` 提取文本块坐标：

1. **多栏检测**：`_detect_columns()` 按文本块 X 坐标聚类，X 间距 > 页宽 15% 识别为分栏
2. **阅读序重排**：`_sort_by_reading_order()` 先按列、再按 Y 坐标排序
3. **标题推断**：`_block_to_markdown()` 根据字号和粗体推断 `#`/`##`/`###` 层级
4. **降级链路**：PyMuPDF 不可用 → 自动回退到 MarkItDown

### 面试话术

> "PDF 解析层面，我们用 PyMuPDF 提取文本块的 bounding box 坐标，通过 X 坐标聚类检测多栏布局，按列优先、从上到下的阅读顺序重排文本块，解决了传统按行读取导致的语义错乱。同时根据字号和粗体信息推断标题层级，输出结构化 Markdown。"

## 4. 三层语义切分 ✅ 已实现

### 实现详情

```
src/libs/splitter/structure_splitter.py        ← 新增 331 行
src/libs/splitter/splitter_factory.py          ← 注册 "structure" provider
config/settings.yaml → ingestion.splitter: "structure"
```

**`StructureSplitter` 完整实现三层策略：**

**Layer 1 — `_split_by_structure()`：结构感知切分**
- Markdown 标题（`#`~`######`）处开启新段
- 代码块（` ``` `）整段保留，不截断
- 表格行（`|...|`）整段保留
- 空行作为段落边界

**Layer 2 — `_merge_incoherent()`：语义连贯性合并**
- 过短 chunk（< `chunk_size × 0.3`）与相邻合并
- 尾部以冒号/逗号/分号结尾（`_INCOMPLETE_ENDINGS`）→ 与下一段合并
- 下一段首字母小写（mid-sentence continuation）→ 合并

**Layer 3 — `_balance_lengths()`：长度平衡**
- 超长 chunk（> `chunk_size × 1.5`）按段落 → 句子 → 硬切再拆分
- 拆后再次扫描，过短残片与相邻合并
- `_apply_overlap()`：按 `chunk_overlap` 字符附加前段尾部

### 面试话术

> "分块采用自研的三层结构感知策略：第一层在 Markdown 标题处切分，表格和代码块整段保留不截断；第二层做语义连贯性检查，以冒号结尾或过短的 chunk 自动与相邻内容合并；第三层做长度平衡，超长段按句子拆分，配合 20% chunk overlap 保持块间连续性。"

## 6. 层级标签（章节路径） ❌ 待实现

### 现状

```
src/ingestion/chunking/document_chunker.py → _inherit_metadata()
├── source_path, doc_hash, chunk_index, source_ref, image_refs ✅
└── heading_path ❌ 不存在
```

- `MetadataEnricher._extract_title()` 只取首个标题，不维护层级栈
- 检索时无法通过上层标题匹配下层内容

### 优化方案

**在 `split_document()` 中维护标题栈，切分后为每个 chunk 附加 `heading_path`：**

```python
heading_stack: List[Tuple[str, int]] = []

for chunk in chunks:
    headings = re.findall(r'^(#{1,6})\s+(.+)$', chunk.text, re.MULTILINE)
    for level_marks, title in headings:
        level = len(level_marks)
        while heading_stack and heading_stack[-1][1] >= level:
            heading_stack.pop()
        heading_stack.append((title, level))
    chunk.metadata["heading_path"] = " > ".join(h[0] for h in heading_stack)
```

### 改动范围

| 文件 | 改动 |
|------|------|
| `src/ingestion/chunking/document_chunker.py` | `split_document()` 增加标题栈扫描 |

### 优化方案

```python
import re
from datetime import datetime
from pathlib import Path

def _detect_content_type(text: str) -> str:
    if re.search(r'^\|.+\|$', text, re.MULTILINE):
        return "table"
    if re.search(r'^```', text, re.MULTILINE):
        return "code"
    if re.search(r'^\d+\.\s|^[-*]\s', text, re.MULTILINE):
        return "list"
    return "text"

# 在 _inherit_metadata() 中增加：
chunk_metadata["content_type"] = _detect_content_type(chunk_text)
chunk_metadata["file_name"] = Path(source_path).stem
chunk_metadata["file_ext"] = Path(source_path).suffix
chunk_metadata["ingested_at"] = datetime.now().isoformat()
```

## 9. Chunk 大小联动 LLM 窗口 ⚠️ 偏保守

### 现状

```yaml

## 11. 查询改写 / HyDE ❌ 全新需求

### 现状

```
src/core/query_engine/query_processor.py
├── jieba 分词 + 停用词过滤 ✅
├── filter 语法解析 (key:value) ✅
└── 无 Query Rewriting / HyDE / Multi-Query ❌
```

- 用户提问直接拿原始 query 做 embedding + BM25，**不做任何改写**
- 模糊/口语化问题（如"九洲搞什么的"）检索效果差
- 无 HyDE（Hypothetical Document Embedding）增强

### 优化方案

**方案 A：LLM Query Rewriting**

```python
class QueryRewriter:
    """用 LLM 将用户口语化问题改写为检索友好的规范表述。"""
    
    def rewrite(self, query: str) -> List[str]:
        prompt = f"将以下问题改写为 2-3 个语义等价但更适合知识库检索的表述：\n{query}"
        # 返回多个改写版本，分别检索后合并
        ...
```

**方案 B：HyDE（推荐，与 A 可叠加）**

```python
class HyDEExpander:
    """生成假想文档片段，用其 embedding 做检索。"""
    
    def expand(self, query: str) -> str:
        prompt = f"请写一段可能回答以下问题的文档片段：\n{query}"
        hypothetical_doc = self.llm.generate(prompt)
        return hypothetical_doc  # 用这段文本做 embedding 检索
```

### 改动范围

| 文件 | 改动 |
|------|------|
| `src/core/query_engine/query_rewriter.py` | **新建**：LLM 改写 + HyDE |
| `src/core/query_engine/hybrid_search.py` | `search()` 中集成 rewriter |
| `config/settings.yaml` | `retrieval.query_rewrite: true` |

### 面试话术

> "查询端做了两层优化：一是 LLM Query Rewriting，将口语化问题改写为 2-3 个规范表述分别检索再合并；二是 HyDE，让 LLM 生成假想文档片段，用其 embedding 做语义检索，解决 query-document 语义鸿沟问题。"

## 面试回答模板（三段论）

### ① 讲挑战

> 我们项目处理多格式文档（PDF 多栏排版、扫描件、PPT 图文混排），主要挑战：
> 1. PDF 双栏排版传统工具按行读取导致语义错乱
> 2. 扫描件 OCR 对表格结构的还原能力差
> 3. 固定长度切分破坏语义完整性，表格/代码被截断
> 4. 口语化提问与知识库文档之间存在语义鸿沟

### ② 讲方案

> **解析层面**：PDF 用 PyMuPDF 坐标聚类做多栏检测和阅读序重排，扫描页自动走 PaddleOCR + Tesseract 双引擎 OCR；PPT 用 LibreOffice 渲染 + Vision LLM 理解幻灯片内容。
>
> **分块层面**：自研三层结构感知切分——标题/表格/代码块感知切分 → 语义连贯性合并 → 长度平衡。每个 chunk 打层级标签（章节路径）、内容类型和来源元数据。
>
> **检索层面**：Hybrid Search（Dense + BM25 + RRF Fusion），查询端做 LLM 改写 + HyDE 扩展，检索后用 LLM Rerank 精排。元数据标签支持按文件类型、时间等维度过滤。

### ③ 讲效果

> chunk 大小通过 RAGAS 评估实验调优。BM25 增量更新解决了大规模知识库的索引重建瓶颈。离线流程用解析失败率、chunk 长度分布等指标持续监控。整体 faithfulness 和 answer_relevancy 相比基线提升显著。
