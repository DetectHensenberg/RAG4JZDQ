## 1. PDF 多栏排版 — 版面分析 ❌

### 现状

```
src/libs/loader/pdf_loader.py
├── MarkItDown.convert() → 按文本流顺序提取
├── PyMuPDF 仅用于图片提取
└── 无 Layout Analysis
```

- `PdfLoader` 使用 **MarkItDown**（Microsoft 开源工具），内部走 `pdfminer` 或类似引擎
- 按文本流从上往下读取，**不识别多栏布局**
- 双栏 PDF 会把左栏第一行和右栏第一行拼到一起，语义完全错乱

### 优化方案

**方案 A（推荐）：引入 MinerU 作为 PDF 解析后端**

```
pip install magic-pdf[full]
```

- MinerU 内置版面分析模型，能识别多栏、表格、图片区域
- 输出结构化 Markdown，保留版面逻辑顺序
- 替换 `PdfLoader` 中的 MarkItDown 调用，对 `.pdf` 文件走 MinerU

**方案 B：引入 Marker**

```
pip install marker-pdf
```

- Marker 同样具备版面分析能力，更轻量
- 适合不想引入大模型依赖的场景

### 改动范围

| 文件 | 改动 |
|------|------|
| `src/libs/loader/pdf_loader.py` | 新增 `_parse_with_mineru()` 方法，在 `load()` 中优先调用 |
| `config/settings.yaml` | 新增 `ingestion.pdf_parser: "mineru"` 配置项 |
| `src/libs/loader/loader_factory.py` | 可选：根据配置选择 MarkItDown 或 MinerU |

### 面试话术

> "PDF 解析层面，我们针对多栏排版引入了版面分析技术（MinerU），先识别文档物理布局——左栏、右栏、表格、页眉页脚——再按逻辑结构提取内容，解决了传统 PyPDF 按行读取导致的语义错乱问题。"

## 2. OCR 表格/代码还原 ❌

### 现状

```
src/libs/loader/pdf_loader.py
├── MarkItDown 对扫描版 PDF 无 OCR 能力
├── PyMuPDF 仅提取嵌入图片
├── ImageCaptioner 用 Vision LLM 描述图片（非 OCR）
└── 无表格结构还原、无代码块识别
```

- 扫描版 PDF 如果 MarkItDown 提取不到文本，会得到**空内容**
- 没有 OCR 管线来处理扫描件
- 表格被 OCR 后会丢失结构（串成一行）
- 代码块 OCR 后缩进丢失

### 优化方案

**方案：MinerU 自带 OCR + 区域检测（与 #1 合并）**

- MinerU 内置 PaddleOCR + 版面分析
- 自动检测区域类型（文字/表格/代码/图片）
- 表格区域输出结构化 Markdown 表格
- 如果不用 MinerU，可独立引入 `paddleocr` + `ppstructure`

### 改动范围

| 文件 | 改动 |
|------|------|
| `src/libs/loader/pdf_loader.py` | 在 `load()` 中检测扫描版 PDF（文本长度 < 阈值），走 OCR 路径 |
| `requirements.txt` | 添加 `magic-pdf[full]` 或 `paddleocr` |

### 面试话术

> "扫描件用 PaddleOCR 配合版面分析，先检测区域类型（文字/表格/代码/图片），表格区域做专门的表格识别保留结构化格式，代码块保持换行和缩进。"

## 4. 三层语义切分 ⚠️

### 现状

```
src/libs/splitter/recursive_splitter.py
├── LangChain RecursiveCharacterTextSplitter
├── 分隔符：\n\n → \n → ". " → " " → ""
├── chunk_size=1000, chunk_overlap=200
└── 无语义连贯性检查，无结构感知
```

- 只有**单层分隔符递归**，本质是"带分隔符优先的固定长度切分"
- ❌ **不识别 Markdown 标题**：不会在 `# 标题` 处开启新 chunk
- ❌ **不保护表格/代码块**：会从表格中间切断
- ❌ **不做语义连贯性检查**：不会合并过短 chunk
- ❌ **不做跨页段落合并**

### 优化方案

**引入结构感知的三层切分策略（新增 `StructureSplitter`）：**

**第一层：基于文档结构的规则切分**
- 检测 Markdown 标题（`#`、`##`）作为切分点
- 表格（`|...|`）和代码块（` ``` `）整段保留，不截断
- 列表项、段落换行作为自然切分边界

**第二层：语义连贯性检查**
- 扫描相邻 chunk，如果上一个以冒号结尾或内容未完结，合并
- 过短 chunk（< chunk_size * 0.3）与相邻 chunk 合并
- 可选：用 Embedding 余弦相似度判断相邻 chunk 语义连贯性

**第三层：长度平衡**
- 过长 chunk 按次级标题/段落再拆分
- 最终确保每个 chunk 在 `[chunk_size * 0.5, chunk_size * 1.5]` 范围内

### 改动范围

| 文件 | 改动 |
|------|------|
| `src/libs/splitter/structure_splitter.py` | **新建**：三层切分策略实现 |
| `src/libs/splitter/splitter_factory.py` | 注册 `"structure"` provider |
| `config/settings.yaml` | `ingestion.splitter: "structure"` |
| `tests/unit/test_structure_splitter.py` | **新建**：切分策略单测 |

## 6. 层级标签（章节路径） ❌

### 现状

```
src/ingestion/transform/metadata_enricher.py
├── _extract_title(): 从首行/首标题提取
├── _extract_summary(): 前3句
├── _extract_tags(): 正则提取关键词
└── 无层级路径（如 "报销政策 > 差旅报销"）
```

- `MetadataEnricher` 提取 `title`、`summary`、`tags`
- **不维护文档层级栈**：不知道当前 chunk 属于哪个章节层级
- 检索时无法通过上层标题匹配到下层内容

### 优化方案

**在 `DocumentChunker` 中维护标题栈，为每个 chunk 附加 `heading_path`：**

```python
# 在切分前扫描文档，构建标题栈
heading_stack = []  # [("# 总则", 1), ("## 范围", 2)]

for chunk in chunks:
    # 检测 chunk 开头是否有新标题
    headings = re.findall(r'^(#{1,6})\s+(.+)$', chunk.text, re.MULTILINE)
    for level_marks, title in headings:
        level = len(level_marks)
        # 弹出同级或更低级别的标题
        while heading_stack and heading_stack[-1][1] >= level:
            heading_stack.pop()
        heading_stack.append((title, level))
    
    # 记录层级路径
    chunk.metadata["heading_path"] = " > ".join(h[0] for h in heading_stack)
    # 例如: "报销政策 > 差旅报销"
```

### 改动范围

| 文件 | 改动 |
|------|------|
| `src/ingestion/chunking/document_chunker.py` | 在 `split_document()` 中增加标题栈扫描，chunk metadata 添加 `heading_path` |
| `src/ingestion/transform/metadata_enricher.py` | 将 `heading_path` 加入可检索字段 |

### 面试话术

> "分块时维护一个标题层级栈，每个 chunk 都标记了它属于的章节路径，比如'报销政策 > 差旅报销'。检索时这些标题词也参与索引匹配，用户搜'差旅报销'能通过层级标签匹配到相关 chunk。"

### 优化方案

**在切分阶段自动检测 content_type：**

```python
def _detect_content_type(text: str) -> str:
    """检测 chunk 内容类型"""
    if re.search(r'^\|.+\|$', text, re.MULTILINE):
        return "table"
    if re.search(r'^```', text, re.MULTILINE):
        return "code"
    if re.search(r'^\d+\.\s|^[-*]\s', text, re.MULTILINE):
        return "list"
    return "text"
```

**补充源标签：**

```python

## 9. Chunk 大小联动 LLM 窗口 ⚠️

### 现状

```yaml
ingestion:
  chunk_size: 1000    # 约 250-300 tokens（中文）
  chunk_overlap: 200
```

- LLM 上下文窗口：qwen-plus 支持 131K tokens
- 检索 top_k=10，每个 chunk ~300 tokens → 总计 ~3000 tokens
- 当前配置**偏保守但合理**

### 优化方案

- 可调大到 `chunk_size: 1500`、`chunk_overlap: 300`，让每个 chunk 信息更完整
- **建议做 A/B 实验**：用 RAGAS 评估面板对比不同 chunk_size 的 faithfulness/relevancy
- 可在 `settings.yaml` 中参数化，方便实验

### 面试话术

> "chunk 大小通过实验配合 LLM 上下文窗口调优。我们用 RAGAS 评估面板对比了 500/1000/1500 三种 chunk_size 下的 faithfulness 和 answer_relevancy，最终选择了在召回准确率和生成效果之间平衡最好的参数。"

## 面试回答模板（三段论）

### 讲挑战

> 我们项目处理多格式文档（PDF、PPT、Word、扫描件），主要挑战是：
> 1. PDF 多栏排版传统工具解析错乱
> 2. 扫描件 OCR 对表格/代码的还原能力差
> 3. 固定长度切分破坏语义完整性

### 讲方案

> 解析层面：PDF 引入版面分析（MinerU），先检测物理布局再按逻辑结构提取；扫描件用 PaddleOCR 配合区域检测，表格保留结构化格式；PPT 用 Vision LLM 理解幻灯片内容。
>
> 分块层面：三层策略——结构感知规则切分 → 语义连贯性合并 → 长度平衡。chunk overlap 保持连续性。同时给每个 chunk 打层级标签（章节路径）、内容类型和来源元数据。

### 讲效果

> chunk 大小通过 RAGAS 评估实验调优。元数据标签在检索阶段支持按时间、来源、类型过滤，提升召回精度。离线流程用解析失败率、chunk 长度分布等指标持续监控。
