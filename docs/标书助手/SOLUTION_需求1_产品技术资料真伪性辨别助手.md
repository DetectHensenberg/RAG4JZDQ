# 解决方案：需求1 — 产品技术资料真伪性辨别助手

> 基于现有 RAG 技术栈实现，**不引入新框架**，复用已有的摄取管道、混合检索和 LLM 能力。

---

## 一、现有技术栈盘点

| 组件 | 技术 | 复用方式 |
|------|------|----------|
| LLM | DashScope qwen-plus | 参数提取、比对分析、报告生成 |
| Vision LLM | DashScope qwen-vl-plus | 解析产品彩页/检测报告中的图片和表格 |
| Embedding | DashScope text-embedding-v3 (1024d) | 产品知识库语义检索 |
| 向量存储 | ChromaDB | 新建 `products` collection |
| 稀疏检索 | BM25 | 型号/品牌等关键词精确匹配 |
| 检索融合 | RRF + Cross-Encoder | 产品匹配结果重排序 |
| 文档解析 | MarkItDown / LayoutPDF / PPTX Loader | 解析厂家技术资料 |
| 前端 | Vue3 + Element Plus | 新增标书助手页面 |
| API | FastAPI + SSE | 新增 `/api/bid/` 路由组 |

---

## 二、核心设计思路

### 不做 Agentic，做 **工作流驱动**

需求1的本质是三个独立工作流，不需要 Agent 自主决策：

```
工作流A：产品选型（检索匹配）
  用户输入招标技术要求 → 混合检索产品知识库 → 返回匹配产品列表

工作流B：资料参数提取（LLM 结构化提取）
  上传厂家资料 → 摄取管道解析 → LLM 提取技术参数 → 结构化存储

工作流C：参数真伪比对（LLM 比对分析）
  选择产品 + 厂家资料 → LLM 比对参数差异 → 生成偏差报告
```

每个工作流都是 **单次请求 → 确定性处理 → 返回结果**，无需多轮迭代。

## 三、架构设计

### 3.1 数据模型

#### 产品知识库 Collection: `products`

产品知识库与通用知识库 (`default`) 分离，使用独立的 ChromaDB collection。

**摄取时的 metadata schema：**

```python
{
    # 继承自现有 pipeline
    "source_path": "海康威视_DS-2CD2T47G2-L_产品白皮书.pdf",
    "chunk_index": 0,
    "heading_path": "技术规格 > 图像参数",

    # 产品专用 metadata（由 LLM 或规则提取）
    "product_category": "摄像头",        # 产品大类
    "product_vendor": "海康威视",         # 厂家名称
    "product_model": "DS-2CD2T47G2-L",  # 产品型号
    "product_brand": "海康威视",          # 品牌
    "doc_type": "official",              # official=官方资料, vendor=厂家送审资料
    "params_json": "{...}",              # 提取的结构化参数 JSON
}
```

#### 参数结构化格式（动态 KV，不固定 schema）

每个产品的参数字段完全不同（摄像头有分辨率/帧率，服务器有CPU/内存，交换机有端口数/背板带宽），因此采用 **动态 Key-Value 数组**，不强制固定字段：

```json
{
    "product_name": "网络摄像头",
    "model": "DS-2CD2T47G2-L",
    "vendor": "海康威视",
    "category": "摄像头",
    "params": [
        {"name": "分辨率", "value": "400万像素 2688×1520", "unit": "像素"},
        {"name": "帧率", "value": "25", "unit": "fps@400万"},
        {"name": "最低照度", "value": "0.003", "unit": "Lux"},
        {"name": "镜头", "value": "2.8mm/4mm/6mm", "unit": "mm"},
        {"name": "防护等级", "value": "IP67", "unit": ""},
        {"name": "工作温度", "value": "-30~60", "unit": "°C"}
    ]
}
```

**不同品类的参数示例：**

```json
// 服务器
{ "category": "服务器", "params": [
    {"name": "CPU型号", "value": "Intel Xeon Gold 6248R", "unit": ""},
    {"name": "CPU核数", "value": "24", "unit": "核"},
    {"name": "内存容量", "value": "256", "unit": "GB"},
    {"name": "硬盘类型", "value": "SSD NVMe", "unit": ""}
]}

// 交换机
{ "category": "交换机", "params": [
    {"name": "端口数", "value": "48", "unit": "个"},
    {"name": "背板带宽", "value": "598", "unit": "Gbps"},
    {"name": "转发速率", "value": "432", "unit": "Mpps"}
]}
```

#### 参数比对策略：LLM 语义匹配

由于不同厂家对同一参数的命名可能不同（如"分辨率"vs"像素数"，"工作温度"vs"Operating Temperature"），比对时**不依赖字段名精确匹配**，而是交给 LLM 做语义对齐：

1. 将两份参数列表完整传给 LLM
2. LLM 自动识别语义相同的参数对（如"分辨率"↔"像素数量"）
3. 逐对比对数值差异
4. 标注无法匹配的项为"缺失"或"新增"

这正是用 LLM 而非规则引擎做比对的核心优势。

### 3.2 工作流详细设计

#### 工作流A：智能产品选型

```
┌──────────────────────────────────────────────────────────────┐
│  用户输入：招标技术要求（文本或上传招标文件片段）              │
│  例："摄像头，分辨率不低于400万像素，支持IP67防护"            │
├──────────────────────────────────────────────────────────────┤
│  Step 1: 查询预处理（现有 QueryProcessor）                    │
│    → 提取关键词：摄像头、400万像素、IP67                      │
├──────────────────────────────────────────────────────────────┤
│  Step 2: 混合检索（现有 HybridSearch）                        │
│    → collection = "products"                                  │
│    → metadata_filter: { "doc_type": "official" }             │
│    → Dense(语义) + BM25(型号/品牌) → RRF → Rerank            │
├──────────────────────────────────────────────────────────────┤
│  Step 3: LLM 结果整理（新增 Prompt）                          │
│    → 从检索结果中提取：厂家、型号、关键参数、匹配度           │
│    → 输出结构化的产品推荐列表                                 │
├──────────────────────────────────────────────────────────────┤
│  输出：产品推荐列表（表格形式）                               │
│  ┌─────────┬────────────────┬──────────┬──────────┐          │
│  │ 厂家    │ 型号           │ 关键参数 │ 匹配度   │          │
│  │ 海康威视│ DS-2CD2T47G2-L │ 400万/IP67│ 95%     │          │
│  │ 大华    │ IPC-HFW5449T1  │ 400万/IP67│ 90%     │          │
│  └─────────┴────────────────┴──────────┴──────────┘          │
└──────────────────────────────────────────────────────────────┘
```

**实现方式**：复用现有 `HybridSearch`，仅需切换 collection 为 `products` 并配合新 prompt。

#### 工作流B：厂家资料参数提取

```
┌──────────────────────────────────────────────────────────────┐
│  用户操作：上传厂家技术资料（PDF/PPTX/DOCX）                  │
│  标记：doc_type = "vendor"                                    │
├──────────────────────────────────────────────────────────────┤
│  Step 1: 文档摄取（现有 IngestionPipeline）                   │
│    → collection = "products"                                  │
│    → 解析文档、提取图片、切分 chunk                           │
├──────────────────────────────────────────────────────────────┤
│  Step 2: LLM 参数提取（新增 Transform 步骤）                  │
│    → 遍历 chunks，用 LLM 提取结构化技术参数                  │
│    → Prompt: "从以下文本中提取产品技术参数，输出JSON格式"     │
│    → 将提取的参数写入 chunk.metadata["params_json"]           │
├──────────────────────────────────────────────────────────────┤
│  Step 3: 参数汇总                                             │
│    → 合并所有 chunk 提取的参数为完整产品参数表                │
│    → 存储到 SQLite (product_params 表)                        │
├──────────────────────────────────────────────────────────────┤
│  输出：结构化的产品参数表                                     │
└──────────────────────────────────────────────────────────────┘
```

**实现方式**：复用现有 `IngestionPipeline`，新增一个 `ParamExtractor` transform 步骤。

#### 工作流C：参数真伪比对

```
┌──────────────────────────────────────────────────────────────┐
│  用户操作：选择一个已提取参数的厂家资料                       │
│  + 指定要比对的官方产品（从知识库选择）                       │
├──────────────────────────────────────────────────────────────┤
│  Step 1: 获取两份参数                                         │
│    → A = 厂家资料提取的参数（vendor）                         │
│    → B = 知识库中的官方参数（official）                       │
├──────────────────────────────────────────────────────────────┤
│  Step 2: LLM 参数比对（核心）                                 │
│    → Prompt：将 A 和 B 的参数逐项比对                        │
│    → 识别：一致 / 偏差 / 缺失 / 新增                        │
│    → 对偏差项给出风险等级和说明                               │
├──────────────────────────────────────────────────────────────┤
│  Step 3: 生成比对报告（SSE 流式输出）                         │
│    → Markdown 表格 + 风险标注                                 │
│    → 偏差项标红，附带原文定位                                 │
├──────────────────────────────────────────────────────────────┤
│  输出：偏差报告                                               │
│  ┌──────────┬──────────┬──────────┬──────┬──────────┐        │
│  │ 参数项   │ 官方值   │ 送审值   │ 状态 │ 风险等级 │        │
│  │ 分辨率   │ 400万    │ 400万    │ ✅   │ -        │        │
│  │ 帧率     │ 25fps    │ 30fps    │ ⚠️偏差│ 中      │        │
│  │ 工作温度 │ -30~60°C │ -40~65°C │ ⚠️偏差│ 高      │        │
│  │ 防护等级 │ IP67     │ -        │ ❌缺失│ 高      │        │
│  └──────────┴──────────┴──────────┴──────┴──────────┘        │
└──────────────────────────────────────────────────────────────┘
```

**实现方式**：纯 LLM 任务，传入两份参数 JSON，用专门的 prompt 驱动比对，SSE 流式输出。

## 四、Skills 集成

将**领域知识和 Prompt 模板**从代码中剥离到 Skill，后续新增品类只需更新 Skill references，不改代码。

### 已创建的 Skills

| Skill | 位置 | 用途 |
|-------|------|------|
| `bid-param-extractor` | `.github/skills/bid-param-extractor/` | 从任意品类文档中提取结构化参数 |
| `bid-param-comparator` | `.github/skills/bid-param-comparator/` | 比对两份参数，生成偏差报告 |

### bid-param-extractor 结构

```
bid-param-extractor/
├── SKILL.md                          ← 工作流 + 动态KV格式说明
├── references/
│   ├── extraction_prompt.md          ← LLM 参数提取 prompt 模板
│   └── category_examples.md          ← 各品类典型参数参考
└── scripts/
    └── merge_params.py               ← 多chunk参数合并脚本
```

**运行时调用**：`ParamExtractor` 读取 `extraction_prompt.md` 作为 prompt，调用 LLM 提取参数，再用 `merge_params.py` 合并。

### bid-param-comparator 结构

```
bid-param-comparator/
├── SKILL.md                          ← 工作流 + 语义匹配说明
├── references/
│   ├── comparison_prompt.md          ← LLM 比对 prompt 模板
│   └── risk_rules.md                 ← 风险等级判定规则
└── scripts/
    └── (预留)
```

**运行时调用**：`ParamComparator` 读取 `comparison_prompt.md` 和 `risk_rules.md`，拼接两份参数 JSON 后调用 LLM 生成偏差报告。

### Skills 的价值

- **新增品类零代码** — 只需在 `category_examples.md` 补充示例
- **Prompt 可独立迭代** — 修改 prompt 不需要改 Python 代码
- **风险规则可定制** — 不同行业/客户可维护独立的 `risk_rules.md`
- **Agent 可复用** — 其他 Agent 调用同一 Skill 即可获得相同能力

## 五、关键 Prompt 设计

### 5.1 参数提取 Prompt

```
你是一位专业的产品技术参数提取专家。请从以下产品技术文档片段中提取所有技术参数。

要求：
1. 输出严格的 JSON 格式
2. 参数名称使用中文
3. 保留原始数值和单位，不做转换
4. 如果能识别出产品名称、型号、厂家，也一并提取

输出格式：
{
    "产品名称": "...",
    "型号": "...",
    "厂家": "...",
    "参数": {
        "参数名1": "参数值1",
        "参数名2": "参数值2"
    }
}

文档片段：
{chunk_text}
```

### 5.2 参数比对 Prompt

```
你是一位严谨的产品技术审查专家。请逐项比对以下两份产品技术参数。

【官方参数（来自产品知识库）】
{official_params}

【送审参数（来自厂家提供的技术资料）】
{vendor_params}

请按以下规则逐项比对：
1. ✅ 一致：两份参数数值相同或在合理误差范围内
2. ⚠️ 偏差：送审值与官方值不一致（标注风险等级：低/中/高）
3. ❌ 缺失：官方有但送审资料中未提供
4. ➕ 新增：送审资料中有但官方资料中没有的参数

风险等级判定：
- 高：核心性能参数偏差（如分辨率、功率、认证等级）
- 中：次要参数偏差（如工作温度范围扩大、尺寸略有不同）
- 低：格式/表述差异（如"≥400万"vs"400万像素"）

输出 Markdown 表格 + 总结：

| 参数项 | 官方值 | 送审值 | 状态 | 风险等级 | 说明 |
|--------|--------|--------|------|----------|------|
| ...    | ...    | ...    | ...  | ...      | ...  |

## 审查结论

- 一致：X 项
- 偏差：X 项（高风险 X 项）
- 缺失：X 项
- 新增：X 项

## 风险提示
（列出需要重点关注的偏差项和缺失项）
```

## 八、与 Agentic RAG 的取舍

| 方面 | 工作流方案（本方案） | Agentic RAG |
|------|---------------------|-------------|
| 响应速度 | 3-8秒 | 15-60秒 |
| 可控性 | 每步可预测 | 不确定 |
| LLM 调用次数 | 1-3次/请求 | 5-15次/请求 |
| API 成本 | 低 | 高 3-5倍 |
| 复杂问题处理 | 需要用户拆解 | Agent 自动拆解 |
| 实现复杂度 | 低（复用现有代码） | 高（引入新框架） |

**结论**：需求1的三个工作流都是**结构明确、输入输出确定**的任务，不需要 Agent 自主决策。工作流方案更快、更稳、更省钱。
