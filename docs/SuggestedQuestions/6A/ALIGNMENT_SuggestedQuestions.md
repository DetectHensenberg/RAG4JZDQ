# ALIGNMENT: Suggested Questions (追问推荐)

## 1. 业务背景与原始需求
- **目标**：在用户获得 RAG 回答后，系统自动推荐 3 个相关的后续问题，引导用户深入探索知识库。
- **痛点**：用户在获得初步答案后可能不知道如何进一步提问，推荐追问可以提高平台交互深度与粘性。

## 2. 需求理解与边界确认
- **生成方式**：初步决定采用 **LLM 异步/同步生成**。通过给 LLM 提供 `Query` 和 `Answer`，让其总结出相关联的追问。
- **展示位置**：集成在 `query_knowledge_hub` 的响应结果中，作为一个独立的字段（建议放在 `references` 之后）。
- **数量限制**：固定推荐 3 个问题。
- **语言对齐**：追问语言需与用户提问语言一致（通常为中文）。

## 3. 技术路径分析 (智能决策)
- **方案 A: 实时 LLM 生成 (推荐)**
  - 在生成完 Answer 后，追问一个 Prompt 让 LLM 产出 3 个问题。
  - 优点：相关性极高。
  - 缺点：增加了一次 LLM 调用成本与延迟。
- **方案 B: 向量库相似提问**
  - 从历史高频提问（或反馈好的提问）中检索相似度最高的问题。
  - 优点：低成本、秒回。
  - 缺点：需要积累足够多的历史数据，冷启动困难。

**当前决策**：优先实现 **方案 A**，但在 `settings.yaml` 中增加开关，允许用户根据成本/延迟需求进行配置。

## 4. 疑问澄清 (待确认点)
1. **生成时机**：是在同一个 LLM 请求内输出（需要修改 Prompt 结构），还是在拿到 Answer 后额外请求一次？
   - *权衡*：同一个请求内输出更容易保证上下文一致性，但可能影响流式输出的体验。
2. **UI 承载**：前端通过什么格式解析？（建议使用标准的 `suggested_questions: List[str]` 结构）。

## 5. 项目集成方案
- 修改 `ResponseBuilder` 以承载追问列表。
- 在 `QueryKnowledgeHubTool` 中增加生成逻辑。
- 在 `Settings` 中新增 `suggested_questions_enabled` 开关。
