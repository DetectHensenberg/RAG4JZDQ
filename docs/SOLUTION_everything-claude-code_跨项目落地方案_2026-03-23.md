# SOLUTION - everything-claude-code 跨项目落地方案（待审批）

> 日期：2026-03-23  
> 目标：将 `everything-claude-code` 作为长期 AI 工程底座使用，而非仅服务单一项目

---

## 1. 背景与结论

`everything-claude-code` 的价值不在某一个仓库里的代码，而在它提供了可复用的：

1. `skills`（任务能力包）
2. `rules`（行为约束）
3. `commands`（标准化入口）
4. `hooks`（自动化触发）
5. `agents`（协作角色）

结论：适合采用“分层裁剪 + 渐进接入”，不建议整仓原样全量落地。

---

## 2. 目标形态（跨项目）

## A. 全局层（个人底座）

跨项目通用，放在用户级环境，所有项目可复用：

1. 通用编码规范与审查规则
2. 常用调研/验证/复盘技能
3. 安全扫描与依赖检查能力
4. 标准化命令入口（如 review、qa、release-check）

## B. 项目层（仓库定制）

与业务强相关，放在项目内：

1. 项目专用工作流（RAG、检索、评测、文档产出）
2. 约束性规则（目录结构、命名、测试门禁）
3. 项目私有命令（例如 `run-qa`, `update-context`）
4. 项目专用 hooks（只在该仓库触发）

## C. 团队层（协作治理）

多人协同时启用：

1. PR 验收模板统一
2. 质量门禁（lint/test/security）统一
3. 变更追踪与文档更新规范统一

---

## 3. 采纳策略（先轻后重）

## 阶段 1：最小可用（1-2 天）

1. 仅启用“只读/建议型”能力：调研、总结、评审建议。
2. 不启用自动写入型 hooks，避免误触发。
3. 建立能力白名单：只保留当前最有收益的 5-10 个 skills。

验收标准：

1. 不影响现有开发节奏。
2. 新增能力可被稳定复用。

## 阶段 2：项目增强（3-5 天）

1. 引入项目级 rules（目录、测试、文档更新要求）。
2. 增加项目命令：`docs-sync`、`qa-smoke`、`release-check`。
3. 建立“任务完成后必须更新 context/报告”的自动提醒机制。

验收标准：

1. 关键任务可按统一流程闭环。
2. 回归质量稳定。

## 阶段 3：团队治理（1-2 周）

1. 固化多人协作模板（需求、设计、任务、验收、复盘）。
2. 按角色拆分 agent 责任边界（实现、审查、测试、文档）。
3. 建立周期性审计：淘汰低价值技能，保留高频高收益能力。

验收标准：

1. 多人协作冲突下降。
2. 文档、测试、交付一致性提升。

---

## 4. 当前建议引入清单（优先）

## 高优先级（建议立即）

1. `search-first`：避免重复造轮子
2. `verification-loop` / `qa-tester`：形成可验证闭环
3. `security-review`：接口和输入处理安全兜底
4. `python-patterns` / `python-testing`：统一 Python 代码质量

## 中优先级（建议第二批）

1. `frontend-patterns`（前端任务频繁时）
2. `api-design`（新增接口较多时）
3. `backend-patterns`（复杂服务拆分时）

## 低优先级（暂缓）

1. 仅少量场景使用的重型 skills
2. 会强耦合当前项目目录结构的通用 hooks

---

## 5. 风险与控制

1. 风险：能力过多导致上下文噪音增加  
控制：白名单机制，阶段性审计。

2. 风险：自动化 hooks 误改代码  
控制：前两阶段禁用自动写入 hooks，仅提示。

3. 风险：规则过严拖慢开发效率  
控制：先软约束（提示），再硬约束（门禁）。

4. 输出一版团队协作模板（需求/设计/任务/验收/复盘）。
