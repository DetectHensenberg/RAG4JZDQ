## 一、项目总览

### 1.1 架构概要

```
src/
├── core/               # 核心层：类型定义、配置、查询引擎、追踪
│   ├── types.py        # 全局数据契约 (Document, Chunk, ChunkRecord, RetrievalResult)
│   ├── settings.py     # YAML 配置加载与校验
│   ├── query_engine/   # 检索引擎 (Dense + Sparse + RRF Fusion)
│   ├── response/       # 响应构建
│   └── trace/          # 追踪上下文
├── ingestion/          # 摄取管道
│   ├── pipeline.py     # 6 阶段编排器
│   ├── chunking/       # 文档分块
│   ├── embedding/      # Dense + Sparse 编码
│   ├── storage/        # ChromaDB + BM25 + Image 存储
│   └── transform/      # Chunk 精炼 + 元数据增强 + 图片描述
├── libs/               # 基础设施层（可替换 Provider）
│   ├── llm/            # LLM 工厂 + 多 Provider 支持
│   ├── embedding/      # Embedding 工厂 + 多 Provider 支持
│   ├── loader/         # 文档加载器工厂
│   ├── vector_store/   # 向量存储工厂
│   ├── splitter/       # 文本分割器
│   ├── reranker/       # 重排序器
│   └── evaluator/      # 评估器
├── mcp_server/         # MCP 协议服务
│   ├── server.py       # stdio 入口
│   ├── protocol_handler.py
│   └── tools/          # MCP 工具实现
└── observability/      # 可观测性
    ├── logger.py       # 结构化日志
    ├── dashboard/      # Streamlit Dashboard
    └── evaluation/     # 评估面板
```

### 1.2 技术栈

| 组件 | 技术选型 |
|------|----------|
| LLM | DashScope (qwen-plus) via OpenAI-compatible API |
| Embedding | DashScope (text-embedding-v3, 1024 维) |
| Vision LLM | DashScope (qwen-vl-plus) |
| 向量数据库 | ChromaDB (本地持久化) |
| 稀疏检索 | BM25 (jieba 分词 + 自研索引) |
| 文档解析 | MarkItDown (PDF/DOCX/MD) + python-pptx (PPTX) |
| 前端 | Streamlit Dashboard |
| MCP 协议 | 官方 Python MCP SDK (stdio) |
| 配置 | YAML (config/settings.yaml) |

## 二、架构设计评审

### 2.1 优点 ✅

**1. 分层架构清晰**
- `core/` → `ingestion/` → `libs/` 三层分离，依赖方向正确
- `types.py` 集中定义数据契约，避免模块间耦合
- Factory Pattern 统一创建 LLM、Embedding、Loader、VectorStore

**2. 摄取管道设计合理**
- 6 阶段清晰：Integrity → Load → Chunk → Transform → Encode → Store
- SHA256 文件去重避免重复处理
- 支持 `on_progress` 回调，Dashboard 可实时显示进度
- `PipelineResult` 数据类封装执行结果，可序列化

**3. Hybrid Search 架构完整**
- Dense (Embedding) + Sparse (BM25) 双通道检索
- RRF Fusion 合并结果
- 支持并行检索 (ThreadPoolExecutor)
- 可选 Reranker 二次排序

**4. 可观测性良好**
- TraceContext 贯穿全链路
- 每个 Stage 记录耗时和详细数据
- Streamlit Dashboard 提供可视化界面

**5. 测试覆盖面广**
- 50+ 单元测试文件覆盖所有核心模块
- 测试分层：unit / integration / e2e
- conftest.py 提供共享 fixtures

### 2.2 问题 ❌

#### P0 - 严重（需立即修复）

**1. API Key 硬编码在配置文件中**
- **位置**: `config/settings.yaml:12,25,35`
- **问题**: `api_key: "sk-02c170ed9467429f8834a3f743b4f215"` 直接写在代码仓库中
- **风险**: 如果推送到公共仓库，API Key 泄露
- **建议**: 使用环境变量 `DASHSCOPE_API_KEY`，settings.yaml 中改为 `api_key: ${DASHSCOPE_API_KEY}`，或使用 `.env` 文件

**2. ChromaDB 数据稳定性问题**
- **位置**: `src/libs/vector_store/chroma_store.py`
- **问题**: 进程终止时 HNSW/FTS5 索引频繁损坏（已发生多次）
- **根因**: Windows SQLite 锁机制 + ChromaDB 写入事务未完成时被中断
- **建议**:
  - 添加 `atexit` 注册优雅关闭
  - upsert 后手动触发 `PRAGMA wal_checkpoint(PASSIVE)`
  - 考虑定期自动备份 chroma.sqlite3

**3. BM25 索引与 ChromaDB 数据不同步**
- **位置**: BM25 存储在 JSON 文件，ChromaDB 存储在 SQLite
- **问题**: 两套存储独立运作，没有事务保证，任何一方失败会导致数据不一致
- **建议**: 在 pipeline Stage 6 添加回滚机制，或引入两阶段提交

#### P1 - 重要（应尽快修复）

**4. IngestionPipeline 每次文件创建完整组件栈**
- **位置**: `src/observability/dashboard/pages/knowledge_base.py:121`
- **问题**: `pipeline = IngestionPipeline(settings, collection=collection)` 在文件循环外创建，但每个 IngestionPipeline 初始化都会创建新的 ChromaDB 连接、Embedding 客户端等
- **影响**: 批量处理多个文件夹时，Dashboard 可能多次创建 Pipeline
- **建议**: Pipeline 实例应复用，避免重复初始化

**5. pipeline.py 单文件 602 行，职责过重**
- **位置**: `src/ingestion/pipeline.py`
- **问题**: 编排逻辑 + Trace 记录 + 日志输出全在一个方法中，`run()` 方法超过 400 行
- **建议**: 将每个 Stage 提取为独立方法，`run()` 仅做编排

**6. LLM 层缺少流式输出支持**
- **位置**: `src/libs/llm/openai_llm.py`
- **问题**: `_call_api` 使用同步 `httpx.Client` + 完整响应，无流式支持
- **影响**: 知识库问答页面需要等待完整响应才能显示，用户体验差
- **建议**: 添加 `chat_stream()` 方法，使用 `stream=True` + Server-Sent Events

**7. 根目录临时脚本未清理**
- **位置**: 项目根目录
- **问题**: `check_db.py`, `check_bm25_sync.py`, `check_metadata.py`, `check_schema.py`, `check_vector_storage.py`, `fix_fts.py`, `fix_fts_v2.py`, `fix_chroma_compaction.py`, `rebuild_bm25.py`, `rebuild_bm25_correct.py`, `remove_fts.py` 共 11 个调试脚本散落在根目录
- **建议**: 移至 `scripts/maintenance/` 或删除

#### P2 - 建议改进

**8. Sparse Encoder 未复用分词结果**
- **位置**: `src/ingestion/embedding/sparse_encoder.py`
- **问题**: jieba 分词在 BM25 检索时再次执行，与 ingestion 时的分词不一致（没有保存分词结果）
- **建议**: 将分词结果持久化到 metadata，检索时直接使用

**9. 配置未与代码分离**
- **位置**: `config/settings.yaml`
- **问题**: `collection_name: "knowledge_hub"` 与 Dashboard 中使用的 `"default"` 不一致
- **建议**: 统一 collection 管理，避免配置歧义

**10. vector_store.collection_name vs pipeline collection 参数**
- **位置**: `chroma_store.py` 使用 `settings.vector_store.collection_name`，但 `pipeline.py` 使用构造参数 `collection`
- **问题**: 存在两个 collection 名来源，可能导致数据写入不同 collection
- **建议**: 统一 collection 管理入口

**11. QA 历史持久化方式过于简单**
- **位置**: `src/observability/dashboard/pages/knowledge_qa.py`
- **问题**: 历史记录保存到 `data/qa_history.json`，随着问答增加，JSON 文件会越来越大；没有分页、没有删除功能
- **建议**: 使用 SQLite 存储历史，添加删除/清空功能

**12. Dashboard 页面文件过大**
- **位置**: `knowledge_qa.py` (29KB), `hybrid_search.py` (30KB), `ingestion_traces.py` (21KB)
- **问题**: 单文件超过 700 行，混合了 UI 逻辑、业务逻辑、数据处理
- **建议**: 提取业务逻辑到 `services/` 层，页面只负责 UI 渲染

**13. 缺少连接池管理**
- **位置**: `src/libs/llm/openai_llm.py:220`
- **问题**: 每次 API 调用都 `with httpx.Client(timeout=120.0) as client` 创建新连接
- **建议**: 在类初始化时创建持久化 `httpx.Client`，复用连接池

**14. 图片存储路径使用绝对路径**
- **位置**: `src/ingestion/storage/image_storage.py`
- **问题**: 图片路径存储为绝对路径，换机器或移动项目目录后路径失效
- **建议**: 存储相对路径，运行时再 resolve

## 七、总体评价

**综合评分: 7.5 / 10**

**架构亮点**:
- 模块化设计好，Factory + Strategy 模式运用得当
- 类型系统完善，数据契约清晰
- 可观测性强，Trace + Dashboard 体系成熟
- 测试覆盖面广

**主要风险**:
- ChromaDB 在 Windows 上的稳定性是最大隐患
- API Key 安全性需要立即处理
- 缺少流式输出影响用户体验
- BM25 + ChromaDB 双存储缺乏一致性保障

**总结**: 这是一个架构设计良好的 RAG 系统，模块化程度高，可扩展性强。主要问题集中在工程实践层面（安全性、稳定性、性能优化），而非架构设计。建议优先处理安全和稳定性问题，再逐步优化性能和用户体验。
